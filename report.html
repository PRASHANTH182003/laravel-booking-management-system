1. Abstract

1.1 Overview

The rapid expansion of the global hospitality industry has created a pressing need for reliable, efficient and scalable software solutions that can handle the entire lifecycle of a hotel reservation — from the moment a guest requests a room to the moment they check out and beyond. Manual booking processes, spreadsheet-driven records and legacy property management tools are no longer adequate in an era where guests expect instant confirmation, real-time availability checks and transparent pricing.

This report presents the design, development and implementation of a Common Hotel and Resort Booking Management System — hereafter referred to as Grand Palace Booking System — built using PHP 8.3 and the Laravel 13.x framework. The system provides a centralised, web-based platform that enables hotel staff to manage room inventories, receive and process booking requests, track the live status of every reservation, collect primary guest identification details and generate real-time revenue analytics through an intuitive administrative dashboard.

The application follows the Model-View-Controller (MVC) architectural pattern as enforced natively by Laravel, ensuring a clean separation of concerns between business logic, data persistence and presentation. Key technical highlights include Eloquent ORM for database interaction, automatic unique booking reference generation, soft-delete support for data integrity, AJAX-based room availability checking, database transactions for atomic booking creation, and a responsive Bootstrap 5 front-end interface accessible from any modern web browser.

The system demonstrates how modern PHP frameworks can be leveraged to deliver enterprise-grade functionality with minimal code complexity, making it a practical and maintainable solution for small-to-medium hospitality establishments. The project serves as a working proof of concept that can be extended with online payment gateways, email notifications, multi-property support and mobile applications in future iterations.

Keywords: Hotel Booking System, Laravel, PHP, MVC Architecture, Eloquent ORM, Booking Management, Room Availability, Reservation Tracking, Bootstrap 5, MySQL, Hospitality Software.

2. Objective

2.1 Primary Objectives

The core objectives driving the development of this booking management system are as follows:

  1. Centralise Reservation Management: Provide a single, unified platform where all booking requests — regardless of how they arrive — are recorded, tracked and managed, eliminating fragmented and error-prone manual workflows.
  2. Real-Time Room Availability Tracking: Implement an availability engine that prevents double-booking by checking for date conflicts across all active reservations before accepting a new booking, using overlap-detection logic at the database query level.
  3. End-to-End Booking Lifecycle Management: Support the complete reservation journey — from initial Pending state through Confirmed, Checked-In, Checked-Out and Cancelled — with each status transition recorded with timestamps for full auditability.
  4. Guest Information Collection and Storage: Capture and securely store primary guest identification details (name, ID type, ID number, nationality, contact information) in compliance with standard hospitality check-in requirements.
  5. Financial Overview and Analytics: Offer an administrative dashboard that aggregates booking counts, revenue figures, room occupancy rates and recent activity so that hotel management can make data-informed operational decisions.

2.2 Secondary Objectives

  1. Modular and Maintainable Codebase: Adhere to Laravel's MVC conventions and PSR-4 autoloading standards so that any developer familiar with the framework can understand, extend or maintain the system with minimal onboarding time.
  2. Responsive User Interface: Deliver a clean, professionally styled front-end built on Bootstrap 5 and Font Awesome that works correctly on desktop, tablet and mobile screens without requiring a dedicated mobile application.
  3. Data Integrity and Safety: Use database transactions for multi-table write operations, soft deletes to preserve historical records, and server-side input validation to guard against invalid or malicious data entry.
  4. Scalable Foundation: Design the database schema and application architecture in a way that supports future extensions such as online payment integration, automated email notifications, multi-property management and role-based access control.
  5. Comprehensive Search and Filter: Enable hotel staff to rapidly locate specific bookings by reference number, guest name, status or date range through a server-side filter mechanism on the bookings list page.

Summary Statement: The overarching goal is to replace manual, error-prone booking processes with a reliable, intuitive and technically robust web application that improves operational efficiency and guest service quality for hotel and resort establishments.

3. Introduction

3.1 Problem Statement

Hotels and resorts of all sizes — from boutique properties to large resort chains — share a common operational challenge: managing the continuous flow of room reservations accurately and efficiently. Traditional approaches involving paper ledgers, Microsoft Excel spreadsheets or basic database tools are inherently limited. They offer no real-time availability checking, no automated reference generation, no systematic status tracking and no consolidated revenue reporting. Staff must manually cross-reference multiple records to determine room availability, leading to booking conflicts, guest dissatisfaction and revenue loss.

Furthermore, when a guest checks in, staff must access scattered paper records or multiple disconnected files to verify reservation details and guest identification. This fragmentation slows the check-in process and increases the likelihood of errors. A dedicated, integrated booking management system addresses all of these pain points in one cohesive solution.

3.2 Scope of the Project

The Grand Palace Booking Management System covers the following functional scope:

  • Room Module: Full create, read, update and delete (CRUD) operations for hotel rooms, with fields for room number, type, floor, price per night, maximum capacity, amenities and availability status.
  • Booking Module: Reservation creation with automatic reference number generation, date conflict detection, total cost calculation, and multi-status lifecycle tracking.
  • Guest Module: Capture and storage of primary guest identification and contact information linked to each reservation.
  • Dashboard Module: Real-time statistics including total bookings, status counts, room occupancy and total revenue.
  • Search and Filter: Server-side filtering by status, date range and keyword on the bookings list.

The following are intentionally outside the scope of the current version: online payment processing, automated email/SMS notifications, guest self-service portal, multi-property support, and role-based access control with granular permissions.

4. Methodology

4.1 Software Development Methodology

The project was developed following an Iterative and Incremental development approach — a practical subset of Agile methodology well-suited to student and small-team projects. Development was broken into three focused iterations:

  1. Iteration 1 – Foundation: Laravel project scaffolding, database schema design, migration creation, Eloquent model definition with relationships, and basic route configuration.
  2. Iteration 2 – Core Modules: Implementation of Room CRUD, Booking CRUD with date conflict detection, booking reference generation, guest information capture, and transaction safety.
  3. Iteration 3 – UI and Analytics: Blade template layout construction, Bootstrap 5 responsive UI, dashboard with statistics, search/filter functionality, status update workflows and final testing.

Each iteration concluded with a functional review to ensure features met requirements before moving forward. This approach allowed rapid detection and correction of design issues early in the development cycle.

4.5 Module Design

The system is divided into four logically independent but interconnected modules:

4.5.1 Room Management Module

Manages the hotel's room inventory. Each room is defined by its number, type, price, capacity, floor, status and list of amenities stored as a JSON array. The module supports full CRUD operations. The isAvailableForDates() method on the Room model performs a compound SQL query to detect any overlap between an incoming booking's date range and existing active reservations for that room, covering all three overlap scenarios: the new booking starts within an existing booking, the new booking ends within one, or the new booking completely encompasses one.

4.5.2 Booking Management Module

The central module of the system. It orchestrates the entire reservation process: receiving a booking request with room selection, date range, guest count, payment method and primary guest details; validating all inputs; checking room availability; calculating the total cost; creating the Booking record and the associated Guest record inside a single database transaction to ensure atomicity; and auto-generating a unique booking reference.

4.5.3 Guest Information Module

Each booking is associated with one or more guest records. At minimum, a primary guest (marked with is_primary = true) is required, capturing full name, ID type, ID number, nationality, phone and email. The design allows for additional non-primary guests to be added to the same booking in future iterations.

4.5.4 Dashboard and Analytics Module

Provides a real-time management overview. The dashboard() controller method executes multiple aggregate queries — total bookings, counts by status, total revenue from confirmed and checked-out bookings, available and occupied room counts, and a monthly revenue breakdown — and passes the results to the dashboard Blade template for visual display.

4.6 Booking Status Workflow

Every reservation progresses through a well-defined state machine. The five possible states and their transitions are illustrated below. Status changes are controlled through the updateStatus() controller method, which also triggers side effects such as updating room status and recording timestamps.

PENDING
→
CONFIRMED
→
CHECKED IN
→
CHECKED OUT
Any state  →  CANCELLED   (with mandatory cancellation reason)

Table 4.3 – Status Transition Side Effects

Transition ToSide Effects Applied Automatically
ConfirmedSets confirmed_at = now(), sets payment_status = 'paid', sets room.status = 'occupied'
Checked InNo automatic side effects; room already marked occupied at confirmation
Checked OutSets room.status = 'available' (room freed for new bookings)
CancelledSets cancelled_at = now(), stores cancellation_reason, sets room.status = 'available'

This state machine design ensures that room availability and booking records are always consistent with each other. A room can only be returned to "available" status through a proper system transition, preventing accidental double-booking.

5.3 Room Model – app/Models/Room.php

namespace App\Models; use Illuminate\Database\Eloquent\Model; use Illuminate\Database\Eloquent\SoftDeletes; class Room extends Model { use SoftDeletes; protected $fillable = [ 'room_number','room_type','description', 'price_per_night','capacity','floor', 'status','amenities','image', ]; protected $casts = [ 'amenities' => 'array', // JSON → PHP array auto-cast 'price_per_night' => 'decimal:2', ]; // Relationships public function bookings() { return $this->hasMany(Booking::class); } // Scopes public function scopeAvailable($query) { return $query->where('status', 'available'); } /** * Checks if this room has any booking overlap for the requested dates. * Covers 3 overlap cases: new booking starts inside, ends inside, * or fully encompasses an existing booking. */ public function isAvailableForDates($checkIn, $checkOut): bool { return !$this->bookings() ->whereIn('status', ['confirmed','checked_in','pending']) ->where(function ($query) use ($checkIn, $checkOut) { $query->whereBetween('check_in_date', [$checkIn, $checkOut]) ->orWhereBetween('check_out_date',[$checkIn, $checkOut]) ->orWhere(function ($q) use ($checkIn, $checkOut) { $q->where('check_in_date', '<=', $checkIn) ->where('check_out_date', '>=', $checkOut); }); })->exists(); } }

class BookingController extends Controller { /** List all bookings with filters and stats */ public function index(Request $request) { $query = Booking::with(['user', 'room'])->orderBy('created_at', 'desc'); if ($request->filled('status')) $query->where('status', $request->status); if ($request->filled('from_date')) $query->where('check_in_date', '>=', $request->from_date); if ($request->filled('to_date')) $query->where('check_out_date', '<=', $request->to_date); if ($request->filled('search')) { $query->where('booking_reference','like','%'.$request->search.'%') ->orWhereHas('user', fn($q) => $q->where('name','like','%'.$request->search.'%')); } $bookings = $query->paginate(15); $stats = [ 'total' => Booking::count(), 'pending' => Booking::where('status','pending')->count(), 'confirmed' => Booking::where('status','confirmed')->count(), 'checked_in' => Booking::where('status','checked_in')->count(), 'cancelled' => Booking::where('status','cancelled')->count(), ]; return view('bookings.index', compact('bookings','stats')); } /** Store new booking – validates, checks availability, calculates total */ public function store(Request $request) { $request->validate([ 'room_id' => 'required|exists:rooms,id', 'check_in_date' => 'required|date|after_or_equal:today', 'check_out_date' => 'required|date|after:check_in_date', 'guests' => 'required|integer|min:1', 'payment_method' => 'required|in:credit_card,debit_card,cash,bank_transfer', 'guest_name' => 'required|string|max:100', 'guest_email' => 'required|email|max:100', // ... (additional validation rules) ]); $room = Room::findOrFail($request->room_id); if (!$room->isAvailableForDates($request->check_in_date, $request->check_out_date)) return back()->withErrors(['room_id' => 'Room not available for selected dates.']); $nights = Carbon::parse($request->check_in_date)->diffInDays($request->check_out_date); $total = $room->price_per_night * $nights; DB::beginTransaction(); try { $booking = Booking::create([ 'user_id' => Auth::id(), 'room_id' => $room->id, 'check_in_date' => $request->check_in_date, 'check_out_date'=> $request->check_out_date, 'guests' => $request->guests, 'total_amount' => $total, 'status' => 'pending', 'payment_method'=> $request->payment_method, 'payment_status'=> 'unpaid', ]); Guest::create([ 'booking_id' => $booking->id, 'full_name' => $request->guest_name, 'email' => $request->guest_email, 'is_primary' => true, // ... other guest fields ]); DB::commit(); return redirect()->route('bookings.show', $booking) ->with('success','Booking created! Ref: '.$booking->booking_reference); } catch (\Exception $e) { DB::rollBack(); return back()->withErrors(['error' => 'Failed to create booking.']); } } /** Update booking status with side effects */ public function updateStatus(Request $request, Booking $booking) { $data = ['status' => $request->status]; if ($request->status === 'confirmed') { $data['confirmed_at'] = now(); $data['payment_status'] = 'paid'; $booking->room->update(['status' => 'occupied']); } if ($request->status === 'cancelled') { $data['cancelled_at'] = now(); $data['cancellation_reason'] = $request->cancellation_reason; $booking->room->update(['status' => 'available']); } if ($request->status === 'checked_out') { $booking->room->update(['status' => 'available']); } $booking->update($data); return back()->with('success','Status updated to '.ucfirst($request->status)); } }

class RoomController extends Controller { public function index(Request $request) { $query = Room::withCount('bookings'); if ($request->filled('type')) $query->where('room_type', $request->type); if ($request->filled('status')) $query->where('status', $request->status); return view('rooms.index', ['rooms' => $query->paginate(12)]); } public function store(Request $request) { $request->validate([ 'room_number' => 'required|string|unique:rooms', 'room_type' => 'required|in:single,double,suite,deluxe,family', 'price_per_night' => 'required|numeric|min:1', 'capacity' => 'required|integer|min:1', 'amenities' => 'nullable|array', 'image' => 'nullable|image|mimes:jpeg,png,jpg|max:2048', ]); $data = $request->except('image'); if ($request->hasFile('image')) $data['image'] = $request->file('image')->store('rooms', 'public'); Room::create($data); return redirect()->route('rooms.index')->with('success','Room added successfully.'); } /** AJAX endpoint – returns available rooms for a date range */ public function checkAvailability(Request $request) { $availableRooms = Room::available() ->when($request->room_type, fn($q) => $q->where('room_type', $request->room_type)) ->get() ->filter(fn($room) => $room->isAvailableForDates( $request->check_in_date, $request->check_out_date )); return response()->json($availableRooms); } }

7. Conclusion

7.1 Summary

This project successfully demonstrates the design and implementation of a fully functional, production-ready Hotel and Resort Booking Management System using PHP 8.3 and the Laravel 13.x framework. The system achieves all primary and secondary objectives outlined at the beginning of this report.

The application provides hotel management staff with a powerful, centralised platform to handle the complete lifecycle of every reservation — from the initial booking request all the way through confirmation, check-in, check-out and, where necessary, cancellation. The five-state status workflow with automatic side effects (room status updates, payment status changes, timestamp recording) ensures that the system's data is always internally consistent, removing the risk of double-bookings or outdated availability information.

The technical implementation showcases the practical power of the Laravel ecosystem: Eloquent ORM relationships and query scopes reduce repetitive database code to clean, expressive PHP; the database transaction wrapper around the booking creation process guarantees atomicity across two related tables; the boot() model hook auto-generates unique reference codes without any controller intervention; and the soft-delete trait preserves historical booking data for auditing purposes even after a record is "deleted" by staff.

The Bootstrap 5 front-end delivers a professional, responsive user interface with an intuitive dark sidebar navigation, colour-coded status badges, gradient statistics cards, and a clean form layout — making the system accessible to non-technical hotel staff without any training overhead.

✓   Atomic booking creation using database transactions

✓   Five-stage booking lifecycle with automatic side-effect management
✓   Soft-delete data preservation for both rooms and bookings
✓   Responsive Bootstrap 5 UI with professional styling
